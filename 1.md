# Repository Alignment Document

## Project
Agency Monorepo Planning Alignment Report

## Document Purpose

This document consolidates and normalizes the analysis conducted across this chat thread into a single planning artifact.

It is intended to help guide repository planning **before implementation**, especially in an agentic coding workflow where AI tools such as Cursor or Windsurf may otherwise make damaging assumptions from incomplete, stale, or overly broad documentation.

This is **not** a coding brief.
This is a **repository alignment and planning correction document**.

---

# 1. Executive Summary

## 1.1 Overall Verdict

The repository is **architecturally strong, strategically sound, and governance-aware**, but it is **not yet fully implementation-safe for agentic coding**.

The strongest part of the project is its **control-plane model**:

- target-state architecture is separated from implementation authority,
- decision state is explicitly tracked,
- dependency approvals are centrally controlled,
- later packages are condition-gated,
- and app-local-first development is used to prevent fake abstraction.

That is the correct direction for a planning-first monorepo.

However, the current document corpus still contains **authority drift** and **stale implementation-shaping detail** in some task/spec/reference files. This means an agentic coding tool could still:

- activate packages too early,
- use outdated version guidance,
- scaffold future-state patterns as current requirements,
- or sequence implementation incorrectly.

## 1.2 Consolidated Planning Verdict

**Current state:** Planning-strong, implementation-unsafe.

**Why:** The governance spine is good, but not all surrounding documents are sufficiently reconciled to prevent AI execution drift.

**Best next step:** Perform one more planning phase focused on **document reconciliation, milestone packetization, and implementation-safe standards**.

# 2. What the Repository Already Gets Right

This section identifies the parts of the planning system that are already strong and should remain intact.

## 2.1 Governance-First Control Plane

The repository’s most valuable planning decision is the separation between different kinds of authority.

The control-plane model distinguishes between:

- **target-state reference**,
- **approved implementation state**,
- **decision state**,
- **dependency authority**,
- **and AI/agent operating constraints**.

This is exactly the right structure for a monorepo being planned for AI-assisted execution.

### Why this is strong

A typical AI coding tool will often confuse:

- “planned eventually,”
- “documented clearly,”
- and “approved to build now.”

Your planning system already attempts to prevent that. That is a major strength and should be preserved as the backbone of the repository.

## 2.2 App-Local-First Package Philosophy

The project correctly rejects premature shared-package creation.

This means:

- logic begins in the app that needs it,
- shared packages are justified only when real reuse emerges,
- and extraction requires explicit approval rather than optimistic design purity.

### Why this is strong

This is more disciplined than generic monorepo advice, and that is good.

In your case, the biggest risk is not under-abstraction. The biggest risk is that an AI coding tool creates **the wrong shared package too early**, which then distorts boundaries and creates long-term maintenance problems.

## 2.3 Website-First Launch Slice

The planning correctly prioritizes:

1. the public agency website first,
2. then the first authenticated internal tool,
3. then additional lanes only when justified by actual operational need.

### Why this is strong

This keeps the build small enough to validate real package needs instead of inventing them up front.

It also provides the right app validation model:

- apps validate packages,
- not the other way around.

## 2.4 Strong Conditional Activation Philosophy

The planning repeatedly uses a condition-gated approach for future packages and apps.

That is correct.

Examples of things that are appropriately treated as conditional or deferred include:

- dedicated API app,
- shared SEO package,
- shared analytics package,
- shared monitoring packages,
- CMS data package,
- notification/email infrastructure,
- platform-style multi-tenant public-site architecture,
- repo-specific MCP tooling,
- and heavier monorepo governance systems.

### Why this is strong

This reduces architectural fantasy.

The project is not trying to “earn scale in advance.” It is trying to **preserve the ability to scale correctly later**.

## 2.5 Domain-Grouped Package Taxonomy

The overall package taxonomy is directionally good.

The domain-grouped package structure, strict dependency flow, and preference for explicit exports and public APIs all align with a maintainable monorepo model.

## 2.6 Good Direction on Analytics, Auth, CMS, and Internal Tools

The planning choices around:

- split analytics handling,
- split auth handling,
- deferred API extraction,
- CRM as likely first authenticated tool,
- and app-local integration of future data systems,

are all strategically sound.

# 3. The Core Problem: Planning Precision Is Behind Planning Quality

The repository’s problem is **not** that the architecture is weak.

The problem is that the project’s **planning precision has not yet fully caught up to its planning philosophy**.

In plain English:

- the repo is thinking about the right things,
- but not all documents are equally safe to hand to an AI implementation agent.

This is the main theme that emerged across all prior responses.

## 3.1 What is happening

There are effectively **two layers** in the corpus:

### Layer A: Newer normalized governance/control docs
These are the strongest documents.
They define what is actually approved, what is deferred, and what dependencies are authoritative now.

### Layer B: Older or broader task/spec/reference docs
Some of these still contain stale details or broader future-state assumptions that are no longer aligned with the control-plane.

A human can usually tell the difference.
An AI coding tool often cannot.

## 3.2 Why this matters

AI coding tools tend to privilege whichever document is:

- more concrete,
- more procedural,
- more specific,
- or more implementation-shaped.

That means a stale or lower-authority file can easily override the *intended* planning hierarchy in practice.

# 5. Main Alignment Failures Found Across the Thread

## 5.1 Authority Drift Between Documents

This is the single biggest issue.

Some documents correctly describe the repository as:

- milestone-gated,
- condition-gated,
- app-local-first,
- dependency-controlled,
- and authority-layered.

Other documents still sound like:

- richer target-state blueprints,
- broad package blueprints,
- or detailed implementation specs from an earlier phase.

### Practical risk
An AI agent may read a lower-authority doc and infer:

- “this package exists now,”
- “this version is approved now,”
- “this toolchain is active now,”
- or “this is the build order.”

## 5.2 Build-Order Drift

One of the clearest conflicts identified earlier was that some target/reference material still implies building shared packages like:

- SEO,
- monitoring,
- compliance,
- analytics,

before or alongside the first website.

That conflicts with the stronger planning posture that says:

- those capabilities may remain app-local in Milestone 1,
- package creation is conditional,
- and shared lanes should not be activated before real multi-consumer proof exists.

### Why this matters
If an agent builds those packages early, it will create structural overhead, fake shared abstractions, and boundary mistakes before the repository has enough real usage to justify them.

## 5.3 Version/Dependency Drift

Another major issue is that some older spec-like documents still encode older pinned versions or older install assumptions, while the newer dependency authority document provides the real baseline.

### Practical risk
An AI tool may:

- install the wrong versions,
- adopt outdated package manager or Turbo patterns,
- or assume future-lane dependencies are already active.

## 5.4 Tooling-Lane Drift

Some documents still imply more advanced tooling or repo infrastructure than the control-plane currently approves.

Examples include:

- repo-specific MCP server/tooling assumptions,
- broader package-guide expectations for packages that should not exist yet,
- or richer platform thinking than current launch scope justifies.

### Why this matters
These details are individually small, but together they increase the chance that an implementation agent builds a “clever future repo” instead of the repo you actually need now.

## 5.5 Missing Milestone Packetization

You have good policy documents.
What is still missing is a **single implementation-safe Milestone 1 packet**.

That packet should say clearly:

- what is approved now,
- what is prohibited now,
- what may remain app-local now,
- what dependencies are authoritative now,
- what documents are reference only,
- and when an agent must stop and escalate.

### Why this matters
Without that packet, the agent must synthesize too much policy on its own.
That is unsafe.

# 7. External Research Consolidation: Monorepos for Marketing Firms / Agency Context

This is where the research was especially useful.

## 7.1 Main conclusion for agency repositories

A marketing/agency repository should **not automatically be modeled like a SaaS platform**.

That is an easy trap.

Many current public examples for multi-tenant Next.js setups are designed for:

- white-label platforms,
- subdomain-based customer products,
- central tenant administration,
- and many domains/customers served by one product model.

That is **not** the same thing as:

- an agency website,
- multiple branded client sites,
- internal operations tools,
- and selective shared capabilities.

## 7.2 Best-fit public-site model for this repo

The current planning conclusion remains correct:

### Default public-site model
- one app per public-facing site,
- shared core/config/ui only where real reuse exists,
- branded sections stay close to the app,
- campaign/marketing logic stays app-local until proven otherwise.

### Why this is right
This is more appropriate for a marketing firm than prematurely building:

- one mega multi-tenant platform,
- white-label public-site infrastructure,
- or centralized content/routing abstraction before operational need exists.

## 7.3 Platform-style multi-tenancy should be explicitly deferred

A future platform/multi-tenant architecture should be treated as:

- conditional later,
- trigger-based,
- and not part of launch planning.

### Trigger examples
Only revisit this if you later have real needs such as:

- many structurally similar client sites,
- centralized domain and tenant routing needs,
- shared tenant administration,
- white-label product behavior,
- or repeated deployment/operational pain across many near-identical sites.

# 9. What Must Be Corrected Before Broad AI Implementation

This section is the most operationally important part of the document.

## 9.1 Correction 1: Freeze Implementation Authority

You need a clear statement that for the current milestone, only a narrow set of documents are implementation-authoritative.

### Recommended implementation-authority bundle
- repository state / implementation approval doc,
- decision status doc,
- dependency authority doc,
- accepted ADRs that do not conflict with those three,
- and one milestone execution packet.

### Why
This prevents lower-authority files from behaving like instructions.

## 9.2 Correction 2: Mark Stale or Broad Task/Spec Files as Non-Authoritative

You need an explicit rule that says:

- task/spec files are planning aids,
- they do not override the control-plane,
- and if they conflict with current governance docs, they must be reconciled before use.

### Why
Without this, agents may still treat old spec detail as live instructions.

## 9.3 Correction 3: Reconcile High-Risk Documents

The earlier analysis identified several categories of high-risk files:

- architecture/reference docs that still imply early creation of later-stage shared packages,
- foundation specs that still contain outdated version assumptions,
- AI/agent instruction docs that imply tooling not currently approved,
- and future-package guide expectations that may encourage premature package creation.

### Why
These are not harmless inconsistencies.
They directly influence how an AI implementation agent will behave.

## 9.4 Correction 4: Create a Milestone 1 Execution Packet

This is a required planning artifact.

### It should define
- approved apps now,
- approved packages now,
- prohibited packages now,
- app-local allowances now,
- exact dependency baseline now,
- launch sequence now,
- and escalation rules.

### Why
This reduces interpretation burden on coding agents.

## 9.5 Correction 5: Add Missing Implementation Standards

Your planning is strong on policy.
It still needs a thin standards layer that explains **how this repo does certain things**.

This should not be a giant handbook.
It should be a small set of repo-specific standards that convert policy into safe implementation behavior.

# 10. Standards That Should Be Added

These standards emerged as the most useful additions after comparing the repo against current external practice.

## 10.1 Next.js Shared Package Consumption Standard

### Purpose
Prevent AI agents from inventing outdated or ad hoc package-consumption patterns in the app layer.

### Should define
- how apps consume local/shared packages,
- how exports should be structured,
- when local packages must be transpiled,
- what not to do,
- and which older workarounds are forbidden.

### Why this matters
Your repo already emphasizes boundaries and shared-package governance, but it should also define the correct Next.js consumption model for those packages.

## 10.2 Turborepo Task and Configuration Standard

### Purpose
Make task orchestration predictable and stop agents from creating inconsistent Turbo patterns.

### Should define
- what belongs in the root `turbo.json`,
- when package-level `turbo.json` is allowed,
- how task inheritance/extension should work,
- environment handling expectations,
- and what configuration duplication is not allowed.

### Why this matters
This converts “we use Turborepo” into a repeatable repo rule set.

## 10.3 Public-Site Architecture Standard

### Purpose
Protect the repository from drifting into platform-style public-site architecture too early.

### Should define
- default = app per public-facing site,
- app-local brand/campaign logic,
- trigger conditions for shared public-site infrastructure,
- trigger conditions for platform-style multi-tenancy,
- trigger conditions for multi-zones.

### Why this matters
It translates the agency/marketing-firm insight into an explicit repository rule.

## 10.4 CMS-Backed Site Standard

### Purpose
Make future CMS-backed work predictable without activating shared CMS infrastructure prematurely.

### Should define
- default CMS lane,
- app-local integration rules,
- preview/draft-mode pattern,
- Studio placement pattern,
- token/secrets handling expectations,
- and trigger conditions for any shared CMS package.

### Why this matters
It turns an approved direction into a safe future operating pattern.

## 10.5 Agentic Implementation Escalation Standard

### Purpose
Translate the governance philosophy into concrete stop rules for coding agents.

### Should define
- when the agent must stop,
- when the agent may proceed app-local,
- when a package proposal is required,
- when version/dependency ambiguity blocks implementation,
- and what document set the agent is allowed to use.

### Why this matters
This is essential for a non-developer-led, AI-assisted build process.

# 12. Practical Correction Sequence

This section converts the analysis into a realistic planning sequence.

### Goal

### Actions
1. Identify implementation-authority docs.

3. Reconcile architecture/reference docs that still imply old build order.
4. Reconcile foundation/spec docs with current dependency authority.

### Actions
1. Create Milestone 1 execution packet.

3. Include exact dependency baseline.
4. Include allowed app-local exceptions.

## Phase 3: Standards Layer

### Goal
Convert policy into repeatable implementation guidance.

### Actions
1. Add Next.js package-consumption standard.
2. Add Turborepo configuration standard.
3. Add public-site architecture standard.

Begin implementation safely.

### Actions
1. Hand only the implementation-authority packet to agents.
