### Vercel Setup Guide (`docs/deployment/vercel-setup.md`)
```markdown
# Vercel Deployment Setup

## Overview

Vercel is the **primary** deployment target for Next.js applications in this monorepo. Use Cloudflare Pages when you need global edge delivery at lower cost or want to reduce vendor lock-in.

## Initial Setup

### 1. Install Vercel CLI

```bash
npm i -g vercel@latest
```

### 2. Login to Vercel

```bash
vercel login
# Opens browser for authentication
```

### 3. Link Monorepo (First Time Only)

From repository root:
```bash
vercel link
? Set up \"~/projects/agency-monorepo\"? [Y/n] y
? Which scope should contain your project? [Your Team]
? Link to existing project? [y/N] n
? What's your project's name? [agency-monorepo]
```

This creates `.vercel/project.json` at root (add to `.gitignore`).

## Per-App Deployment Setup

### Example: CRM App

```bash
cd apps/internal-tools/crm

# Link this specific app to Vercel project
vercel link

? Set up \"~/projects/agency-monorepo/apps/internal-tools/crm\"? [Y/n] y
? Which scope? [Your Team]
? Link to existing project? [y/N] n
? What's your project's name? [agency-crm]
```

### Configure Build Settings

In Vercel Dashboard → Project → Settings:

**Build & Development Settings:**
```
Framework Preset: Next.js
Build Command: cd ../.. && pnpm install && pnpm --filter @agency/crm build
Output Directory: .next
Install Command: (leave empty, handled by root)
```

**Root Directory:**
```
apps/internal-tools/crm
```

**Environment Variables:**
```
DATABASE_URL=postgresql://...
NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_...
CLERK_SECRET_KEY=sk_...
```

## Database Connection (Neon)

### Neon Branching Strategy

Each preview deployment should use a Neon database branch:

```
main branch (production)
  ├── dev-branch (shared development)
  ├── pr-123-branch (PR-specific)
  └── pr-456-branch (PR-specific)
```

### Automatic Branch Creation

Use Vercel integration or GitHub Actions to:
1. Create Neon branch for PR
2. Run migrations on branch
3. Set DATABASE_URL for preview deployment
4. Delete branch when PR closes

## Custom Domains

### Adding a Domain

1. Vercel Dashboard → Project → Domains
2. Enter domain: `crm.youragency.com`
3. Configure DNS:
   - Type: CNAME
   - Name: crm
   - Value: cname.vercel-dns.com

### Domain per Environment

```
Production: crm.youragency.com
Staging: crm-staging.youragency.com
Preview: auto-generated by Vercel
```

## Troubleshooting

### Build Failures

```bash
# Check logs
vercel logs --all

# Rebuild with fresh cache
vercel --force
```

### "Cannot find module" errors
Usually a monorepo workspace issue:
1. Verify `pnpm-workspace.yaml` includes the app
2. Check that dependencies use `workspace:*`
3. Ensure root `pnpm install` was run

### Database Connection Errors
1. Verify IP allowlist includes Vercel IP ranges
2. Check DATABASE_URL is set in Vercel dashboard
3. Test locally with production DATABASE_URL

### Cloudflare Setup Guide (`docs/deployment/cloudflare-setup.md`)
```markdown
# Cloudflare Pages Deployment Setup

## Overview

Cloudflare Pages is the **fallback/secondary** deployment target. Use it when:
- Cost optimization is critical (especially at scale)
- You need true global edge deployment (200+ locations)
- Reducing vendor lock-in from Vercel
- Building static or lightly dynamic sites

## When to Choose Cloudflare over Vercel

| Factor | Vercel | Cloudflare |
|--------|--------|------------|
| Next.js features | Full support | Partial (no API routes in same way) |
| Edge locations | 100+ | 300+ |
| Cost at scale | Higher | Lower |
| Build time limits | 45min (Pro) | 20min (free), 60min (paid) |
| Bandwidth | 1TB/mo (Pro) | Unlimited |
| Functions | Node.js runtime | V8 isolate (Web Standard APIs) |

## Setup Steps

### 1. Install Wrangler CLI

```bash
npm install -g wrangler
```

### 2. Authenticate

```bash
wrangler login
# Opens browser for Cloudflare authentication
```

### 3. Per-App Configuration

Create `wrangler.toml` in app directory:

```toml
name = "agency-crm"
compatibility_date = "2026-04-01"

[build]
command = "pnpm build"

[[env.production]]
name = "agency-crm-prod"

[[env.staging]]
name = "agency-crm-staging"
```

### 4. Environment Variables

Set via Cloudflare Dashboard or CLI:

```bash
wrangler secret put DATABASE_URL
wrangler secret put AUTH_SECRET
```

## Monorepo Considerations

Cloudflare Pages works best with:
- Static site generation (SSG)
- Client-side data fetching
- API routes in separate Worker (see below)

### Architecture: Separate API Worker

For full-stack apps on Cloudflare:

```
apps/internal-tools/crm/
├── frontend/ → Deployed to Cloudflare Pages
└── api/ → Deployed to Cloudflare Workers
```

## Database Connection

**Neon + Cloudflare:**
- Use `@neondatabase/serverless` (compatible with Workers)
- Connection string in `DATABASE_URL` secret

**Supabase + Cloudflare:**
- Use Supabase REST API (not direct Postgres)
- Better fit for Cloudflare's edge model

## Limitations vs Vercel

- No file-system API routes (must use Workers)
- No Image Optimization (use Cloudflare Images)
- Different caching behavior
- No Turborepo remote cache integration (yet)

## Migration from Vercel

1. Ensure app works with `output: 'export'` (static)
2. Or refactor API routes to Cloudflare Workers
3. Update build command in `wrangler.toml`
4. Deploy and verify

### Platform Comparison (`docs/deployment/platform-comparison.md`)
```markdown
# Platform Comparison

## Deployment Strategy

**Primary (Default):** Vercel
- Best Next.js integration
- Preview deployments
- Turborepo native support

**Fallback:** Cloudflare Pages  
- Cost optimization at scale
- True global edge (300+ locations)
- Reduced vendor lock-in

**Long-running workloads:** Railway or Render
- Background jobs
- Services >60 seconds
- Stateful processes

---

## Vercel (Primary)

**Best For:**
- Next.js applications with full feature set
- Serverless functions
- Teams wanting fastest DX
- Preview deployment workflows

**Pros:**
- Zero-config Next.js
- Edge network (100+ locations)
- Preview deployments per PR
- Turborepo integration
- Generous free tier

**Cons:**
- Function timeout limits (10s hobby, 60s pro)
- No long-running processes
- Higher cost at scale
- Vendor lock-in for advanced features

**Use For:**
- Agency website
- Client sites with heavy Next.js features
- Internal tools (lightweight)
- When DX speed matters most

---

## Cloudflare Pages (Fallback)

**Best For:**
- Cost-sensitive deployments
- Maximum global edge coverage
- Static and lightly dynamic sites
- Reducing Vercel lock-in

**Pros:**
- Lower cost at scale
- 300+ edge locations
- Unlimited bandwidth
- Good Next.js App Router support

**Cons:**
- Limited API route patterns (needs Workers)
- No Image Optimization
- Different caching model
- Smaller ecosystem than Vercel

**Use For:**
- Marketing sites
- Content-heavy client sites
- High-traffic portals (cost optimization)
- When you need maximum global reach

---

## Railway (Long-running workloads)

**Best For:**
- Long-running processes
- Background workers
- Services needing >60s execution
- Stateful applications

**Pros:**
- Longer timeouts (unlimited on paid)
- Persistent storage available
- Good database integration
- Simple deployment

**Cons:**
- No edge network
- Manual preview environments
- Separate from your primary deployment

**Use For:**
- Report generation workers
- Email batch sending
- Data import/export jobs
- Heavy computation tasks

---

## Render

**Best For:**
- Full-stack applications (non-Next.js)
- Static sites
- Background workers
- Budget-conscious simple hosting

**Pros:**
- Simple deployment from Git
- Free tier for static sites
- Good for non-Next.js services

**Cons:**
- Slower than Vercel for Next.js
- Cold start times
- No edge network

---

## Decision Matrix

| Requirement | Vercel | Cloudflare | Railway | Render |
|-------------|--------|------------|---------|--------|
| Next.js optimized | ✅ | ⚠️ | ⚠️ | ⚠️ |
| Edge network | ✅ (100+) | ✅ (300+) | ❌ | ❌ |
| Long running (>60s) | ❌ | ❌ | ✅ | ✅ |
| Background jobs | ⚠️ | ⚠️ | ✅ | ✅ |
| Cost at scale | 💰💰 | 💰 | 💰💰 | 💰 |
| Preview deployments | ✅ | ⚠️ | ❌ | ❌ |
| Turborepo integration | ✅ | ❌ | ❌ | ❌ |
| Vendor lock-in risk | Higher | Lower | Medium | Medium |

---

## Platform Selection Guide

### For Client Sites

| Client Need | Recommended Platform |
|-------------|---------------------|
| Standard marketing site | Vercel (default) |
| High traffic, cost-sensitive | Cloudflare |
| Heavy CMS/Sanity integration | Vercel |
| Maximum global performance | Cloudflare |
| Client requests specific platform | Match their preference |

### For Client Portals

| Portal Type | Recommended Platform |
|-------------|---------------------|
| Feature-rich Next.js portal | Vercel |
| Cost-sensitive at scale | Cloudflare |
| Using Supabase backend | Either works |
| Real-time features | Vercel (better SSE/WebSocket support) |

### For Internal Tools

| Tool Type | Recommended Platform |
|-----------|---------------------|
| CRM, project tracker | Vercel |
| Report generator worker | Railway |
| Email batch processor | Railway |
| Cost-sensitive internal tool | Cloudflare |

## Migration Paths

### Vercel → Cloudflare

**When:** Cost optimization needed, or reducing lock-in
**Effort:** Low-Medium
**Steps:**
1. Verify app works with static export or refactor API routes
2. Create `wrangler.toml` configuration
3. Deploy to Cloudflare Pages
4. Update DNS
5. Monitor for issues

### Any Platform → Railway (for long-running)

**When:** Need background workers or >60s execution
**Effort:** Low
**Steps:**
1. Extract long-running logic to separate service
2. Deploy service to Railway
3. Update main app to call Railway service
4. Keep main app on Vercel/Cloudflare

## Environment Variable Mapping

Different platforms handle env vars differently:

| Var Type | Vercel | Cloudflare | Railway |
|----------|--------|------------|---------|
| Production | `Production` env | `production` via wrangler | Production environment |
| Preview | `Preview` env | Manual branch config | Staging environment |
| Local | `.env.local` | `.dev.vars` | `.env` |
| Secrets | Dashboard/CLI | `wrangler secret` | Dashboard |

**Recommendation:** Keep a master env template in 1Password/Bitwarden and sync to each platform.

## Platform-Specific CI/CD

### Vercel (via GitHub Actions)
Already covered in TASK_25 and TASK_27.

### Cloudflare (via GitHub Actions)

Add to `.github/workflows/deploy-cloudflare.yml`:

```yaml
name: Deploy to Cloudflare

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
      - uses: actions/setup-node@v4
        with:
          node-version: 24
          cache: pnpm
      - run: pnpm install
      - run: pnpm --filter @agency/crm build
      - name: Deploy to Cloudflare
        uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          workingDirectory: apps/internal-tools/crm
```
