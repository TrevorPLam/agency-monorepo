# Recommended system

I recommend this pattern:

```text
directories: {phase}-{domain}-{topic}
files:       {local}-{topic}-{doc}[-variant].md
```

Use a controlled document vocabulary so agentic coders see intent immediately:

- `overview` — what this task is, why it exists, what it excludes.
- `spec` — canonical implementation requirements.
- `adr` — decision record with rationale.
- `guide` — procedural implementation or setup instructions.
- `ref` — reference matrix, schema map, env var table, dependency table.
- `checklist` — acceptance, QA, or release checks.
- `prompt` — agent instructions or bounded implementation prompt.
- `plan` — rollout sequence or dependency sequence.

I would also standardize the **internal file order** so every task folder feels identical:

```text
00-{topic}-overview.md
01-{topic}-spec.md
02-{topic}-constraints.md
03-{topic}-adr-{decision}.md
04-{topic}-guide-{result}.md
05-{topic}-qa-checklist.md
06-{topic}-handoff-prompt.md
```

That sequence is not pure Diátaxis, but it is much better for execution control. Diátaxis is still useful as the thinking model behind the docs: `guide` maps to how-to, `ref/spec` maps to reference, `adr` maps to explanation, and `tutorial` should be used only when you truly want a learning walkthrough rather than build instructions. [diataxis](https://diataxis.fr/how-to-guides/)

## Directory strategy

I would **not** broadly consolidate directories just because they are conceptually related. Your repository already treats docs, tools, infra, testing, analytics, experimentation, and lead capture as separate roadmap units, and some of those are still placeholders or conditional packages, which means separation is already doing planning work for you. [ppl-ai-file-upload.s3.amazonaws](https://ppl-ai-file-upload.s3.amazonaws.com/web/direct-files/collection_1e34ddf2-9fab-4254-9e09-6d3c80436807/0ad64119-e2d5-43ba-837d-680eebb55dc0/b3-tools-codemods-00-package.md)

Use this rule instead:

- Keep separate directories when work can be deferred independently.
- Keep separate directories when acceptance criteria differ.
- Keep separate directories when ownership or risk differs.
- Consolidate only when items always ship together and would genuinely share one spec, one checklist, and one delivery gate.

Applied to your examples, I would handle them like this:

- `40-seo`, `41-compliance`, `42-monitoring`: keep separate unless one engineer or agent would always implement them as one package in one sprint.
- `60-auth-internal`, `61-auth-portal`: keep separate, because internal auth and portal auth usually differ in flows, providers, and risk, even if they share concepts. ## Concrete rename model

Here is the naming plan I would actually adopt.

| Current | Better | Best |
|---|---|---|
| `00-foundation/00-root-scaffolding.md` | `00-foundation/00-foundation-spec.md` | `00-foundation/01-foundation-spec.md` plus `00-foundation/00-foundation-overview.md` |
| `10-config-eslint/00-package.md` | `10-config-eslint/00-eslint-spec.md` | `10-config-eslint/01-eslint-spec.md` plus `10-config-eslint/00-eslint-overview.md` |
| `50-data-db/00-package.md` | `50-data-db/00-data-db-spec.md` | `50-data-db/01-data-db-spec.md` plus `50-data-db/00-data-db-overview.md` |
| `a0-docs-agents/00-package.md` | `a0-docs-agents/00-agents-spec.md` | `a0-docs-agents/00-agents-overview.md`, `01-agents-rules-spec.md`, `02-agents-handoff-prompt.md` |
| `80-analytics/00-package.md` | `80-analytics/00-analytics-spec.md` | `80-analytics/00-analytics-overview.md`, `01-analytics-spec.md`, `02-analytics-guide-event-tracking.md`, `03-analytics-qa-checklist.md` |

If you want one single convention to lock in now, I would choose this:

```text
task folders:
{phase}-{domain}-{topic}

files inside each folder:
00-{topic}-overview.md
01-{topic}-spec.md
02-{topic}-constraints.md
03-{topic}-adr-{decision}.md
04-{topic}-guide-{result}.md
05-{topic}-qa-checklist.md
06-{topic}-handoff-prompt.md
```

That gives you priority order, strong searchability, clean scaling, and far better control over how agentic coders interpret each task than `00-spec.md` ever will.
