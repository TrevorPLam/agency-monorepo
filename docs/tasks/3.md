# Repository Gap-Fill Master Plan

## 0. Purpose of this document

This is the consolidated, repository-ready planning brief for the **remaining missing planning units** after:

* the correction checklist is assumed complete,
* the repo-wide control model is assumed to exist,
* and the previously identified first-wave gaps are treated as valid.

This document is for **planning completion**, not implementation authorization.

For every new task family below:

* the task family itself is approved to be written,
* the corresponding app/package/standard is **not** automatically approved to be scaffolded,
* and `REPO-STATE.md`, `DECISION-STATUS.md`, and `DEPENDENCY.md` remain the authority sources for implementation, decision status, and versions.

---

## 1. Research-validated constraints that still hold

These recommendations still validate against current official docs:

* **Next.js 16** removed `next lint`, still supports ESLint in monorepos via `@next/eslint-plugin-next` and `rootDir`, and still supports App Router Route Handlers / Backend-for-Frontend patterns. Next.js also explicitly says its backend capabilities are **not** a full backend replacement. This validates keeping a dedicated API app conditional, not default. ([Next.js][1])
* **React Compiler** is still current, is documented for React 19, and is installed via the Babel plugin integration path. That keeps the React Compiler lane valid, but it should remain controlled rather than silently becoming a universal default. ([React][2])
* **Tailwind CSS v4** is still the current Tailwind generation, and Tailwind still has an official Next.js setup path. This validates keeping Tailwind v4 in the architecture and reinforces that Vite should not be framed as the default frontend lane in a Next-first repo. ([Tailwind CSS][3])
* **Biome** is technically viable in large monorepos and supports big-project / workspace patterns. But that does not override the stronger official alignment between Next.js and ESLint, so the safest planning stance is still ESLint canonical, Prettier canonical, Biome experimental or migration-only. ([Biome][4])
* **Playwright** still gives you isolated browser contexts, parallel execution, and cross-browser coverage across Chromium, Firefox, and WebKit. That keeps Playwright as the correct E2E lane for real app flows, not as a shared package abstraction. ([Playwright][5])
* **Sanity** still supports Next.js through `next-sanity`, App Router visual editing, and embedding Studio as a route inside Next.js. That validates keeping Studio planning conditional but real. ([Sanity.io][6])
* **Changesets** still positions itself as monorepo versioning/changelog tooling and still handles monorepo dependency bumping behavior. That validates keeping Changesets as infra/release workflow, not as a normal config package. ([GitHub][7])
* **Better Auth**, **Clerk Organizations**, and **WorkOS AuthKit / Directory Sync** still validate the narrowed auth lane: Better Auth first for client portals, Clerk first for internal tools, WorkOS only as enterprise escalation. ([Better Auth][8])
* **Plausible** still supports privacy-friendly analytics with custom events and properties, **PostHog** still ties experiments to feature flags, and **Vercel Web Analytics / Speed Insights** remain credible “start simple” monitoring/analytics defaults. That validates keeping analytics, experimentation, attribution, and observability scope narrow at launch. ([Plausible Analytics][9])

These validations do **not** change the planning conclusion. They reinforce it:

* keep the repo lean,
* keep the control model strong,
* add only the genuinely missing planning units,
* and do not invent new package/platform lanes.

## 3. Why these are the only missing task families

These 10 are justified because they cover planning owners that are still absent.

### 3.1 Real missing owners

These are true omissions:

* `apps/client-portal` / client portal planning lane
* Playwright E2E planning lane
* Studio planning lane
* dedicated API planning lane
* client brand-foundation planning lane
* tenant isolation standard
* dependency truth standard
* `apps/client-sites/` family planning lane
* `docs/analytics/` planning lane
* `docs/decisions/` planning lane

### 3.2 Not missing task families — just indexing or cleanup problems

These are **not** new-task problems:

* `01-config-biome-migration`
* `14-config-biome`
* `15-config-vite`
* `16-config-changesets`
* `25-core-testing`
* `26-core-performance`
* `72a-notifications-orchestration`

These already exist as task families and need to be properly indexed in `docs/tasks/README.md`.

### 3.3 Not missing task families — just thin or inconsistent existing tasks

These also do **not** justify new task families:

* security headers duplication
* CI workflow duplication
* CODEOWNERS duplication
* testing sequencing ambiguity
* thin docs/tooling/infra tasks
* overly broad auth/data/analytics promises

Those are correction and expansion tasks inside existing folders.

### 3.4 Not missing yet — future example apps only

These should **not** become separate task families yet:

* `project-tracker`
* `invoicing`
* `reporting`
* `operations-dashboard`

Those remain architecture examples / future internal-tools descendants, not currently justified independent planning lanes.

### 3.5 Important repository conflict to resolve inside the missing tasks

There is one meaningful architecture conflict that must be resolved during gap fill:

* the repo tree models portals under `apps/client-sites/[client]-portal/`
* but the conditional-app table names `apps/client-portal/`

Do **not** let both survive as equally valid patterns.

**Recommended resolution:**

* keep `e10-apps-client-sites-foundation` as the owner of client-site family topology,
* keep `e4-apps-client-portal` as the owner of portal behavior and portal conditions,
* and set the default canonical portal topology to **`apps/client-sites/[client]-portal/`** unless the repo later proves a need for a single shared multi-tenant portal product.

## 4. Required structure for every new task family

Use the existing task system convention in `docs/tasks/1.md`.

### 4.1 Folder pattern

Use:

* `docs/tasks/{task-family}/`

### 4.2 File pattern

Every new task family should use the full 7-file convention:

* `00-{topic}-overview.md`
* `01-{topic}-spec.md`
* `02-{topic}-constraints.md`
* `03-{topic}-adr-{decision}.md`
* `04-{topic}-guide-{result}.md`
* `05-{topic}-qa-checklist.md`
* `06-{topic}-handoff-prompt.md`

### 4.3 Mandatory header fields in every overview/spec

Every new task family must include the universal header fields:

* **State**
* **Trigger**
* **Minimum consumers**
* **Dependencies**
* **Exit criteria**
* **Implementation authority**
* **Version authority**
* **Supersedes / superseded by**

### 4.4 Required cross-links in every task family

Every new task family must point back to:

* `REPO-STATE.md` for implementation authority
* `DECISION-STATUS.md` for status / unresolved decisions
* `DEPENDENCY.md` for version authority
* `ARCHITECTURE.md` for structural alignment
* `docs/AGENTS.md` for AI guardrails
* any directly related task families

### 4.5 State vocabulary

Use only repo-normalized state vocabulary. Do not use vague terms like:

* ready
* future
* later
* can be used
* optional

unless those phrases are explicitly tied to a defined state in `REPO-STATE.md`.
